## Editorial Guardrails (from `docs/RULES/fcp-docs-rules.md`)

These are content rules that should be enforced as part of the migration:

- Lane discipline: Protocol explains theory/semantics; Reference owns fields/status codes; Guides own onboarding and “how to start”.
- Namespace rules: never introduce `fide:` predicates; only `did:fide:` is valid `fide` usage.
- Identifier formatting: refer to Fide IDs as `did:fide:0x...` in prose; bare `0x...` only for raw values (signatures, roots, addresses, tx hashes) and explicit pre-derivation `rawIdentifier`.
- Terminology: “Distributed”, “Fide Context Graph”, “Fide ID”, “Statement”, “Attestation”, “EvaluationMethod”; avoid “Federated/Decentralized”; keep Verdict vs Score distinction.
- Cross-linking: when content mentions entities, ID calculation, or statement structure, link to the owning protocol pages/anchors.
- Examples: placeholders must remain syntactically valid; temporal examples must be RFC 3339 UTC (`...Z`).

## Current State (What Exists Today)

### 1) Three parallel doc surfaces are active

- Protocol pages: `/apps/docs/content/docs/(protocol)/...`
- REST section wrapper page(s): `/apps/docs/content/docs/reference/v1/...`
- SDK authored pages (legacy, to be replaced): `/apps/docs/content/docs/sdks-legacy/...`

Counts (MDX files):

- SDK docs: 17 files
- Protocol docs: 12 files
- API authored docs: 1 file (plus generated OpenAPI operation pages)

### 2) OpenAPI operation pages are generated at build time

Source config in `/apps/docs/src/lib/source.ts`:

- `openapiSource(...)` with:
  - `baseDir: 'reference'`
  - `per: 'operation'`
  - custom naming => `/reference/v1/attestations-post`, `/reference/v1/health-get`

So API operation references are already mostly generated.

### 3) Docs build currently reads a static OpenAPI artifact

OpenAPI input in `/apps/docs/src/lib/openapi.ts`:

- uses `process.env.OPENAPI_URL` or local `apps/docs/openapi.json`

That improves build reliability (no required running local API), but means docs consume serialized schema.

### 4) SDK samples inside OpenAPI are currently static after export

In `/apps/api/src/routes/attestations.ts`:

- `x-codeSamples` includes JS and Python samples.

But exported `/apps/docs/openapi.json` stores `source` as literal strings.
Function-based dynamic sample generators do not survive JSON serialization, so operation-page field edits do not dynamically update SDK examples.

### 5) Content overlap / user confusion risk

- Protocol pages explain lifecycle and concepts.
- SDK pages repeat many implementation-level explanations.
- API pages and SDK docs both discuss “which to use” and usage flows.

Result: multiple places to update when behavior changes.

---

## Problem Statement

The docs are split by *surface type* (protocol vs SDK vs REST) rather than *user task flow*.
This increases maintenance and forces duplication:

- same concepts repeated in protocol + SDK + API wrapper pages
- SDK reference pages have to be kept in sync with API changes
- generated API pages are clean, but surrounded by manually maintained parallel trees

## Target State (What We Should Move To)

## Information Architecture

Keep only 3 top-level concerns:

1. `Protocol` (spec semantics, implementation-agnostic)
2. `Reference` (generated operation pages from OpenAPI)
3. `Guides` (minimal authored onboarding and decision pages)

Proposed shape under `/apps/docs/content/docs`:

- `(protocol)/...` (keep)
- `reference/...` (generated OpenAPI pages; canonical target)
- `guides/quickstart.mdx`
- `guides/signing-auth.mdx`
- `guides/concepts.mdx`
- `guides/choose-sdk-vs-http.mdx`
- `guides/sdks.mdx` (installation, client setup, retries/timeouts, error handling)

Generated SDK reference under `reference/sdk/*` (auto-generated MDX from `packages/fcp/src`).

Naming conventions:

- Keep the folder label as `sdks/js` if retained for legacy routes; in npm ecosystems, “JS” is the umbrella even when code is TypeScript.
- In code tabs, label the SDK tab `TypeScript` if snippets contain TS types/annotations.

Future additions:

- Add a `Next.js` code tab for reference operations if it differs materially (e.g., route handlers).
- Add `guides/integrations/nextjs.mdx` for an end-to-end integration walkthrough.

### SDK strategy

- Do **not** maintain large per-language trees (`sdks/js`, `sdks/python`) as full references.
- Keep SDK examples embedded on operation pages as code tabs.
- If language support expands, generate those tabs from shared templates/functions, not authored page trees.
  - Generate an SDK reference section (searchable) from TypeScript source rather than writing `sdks/js/*.mdx` by hand.

### What stays in each section

- Protocol:
  - entities, schema semantics, lifecycle rules, trust/query model
  - cryptographic and lifecycle theory
  - no endpoint walkthroughs
  - no language-by-language tutorials
  - no HTTP status code or field-by-field request/response docs
- Reference:
  - request/response contract, params, status codes
  - shared component schemas/types (OpenAPI `components.schemas`)
  - code tabs (`HTTP`, `JavaScript SDK`, optional others)
- Guides:
  - short task-oriented docs and “start here” flows

## Migration Plan (Phased)

### Phase 1: IA and navigation cleanup

1. Adopt final path label as `/reference` with no `/api` compatibility redirects (pre-launch decision).
2. Update sidebar/meta to present:
   - Start Here
   - Protocol
   - Reference
3. Remove “SDK Overview” as a parallel reference tree entry (or reduce to one landing page that links into Reference + Guides).
4. Add `guides/sdks.mdx` for SDK setup/config/runtime concerns that do not belong on endpoint pages.

### Phase 2: De-duplicate authored docs

1. Keep protocol pages in `(protocol)` and trim implementation examples that duplicate reference pages.
2. Implement generated SDK reference pages under `reference/sdk/*` (see Phase 2.5).
3. Keep `/sdks-legacy/*` until generated SDK reference exists and redirects are in place.
4. Archive or redirect `/sdks-legacy/*` only after generated SDK reference exists.
5. Preserve only small guide set (quickstart, auth/signing, concepts, choose SDK vs HTTP, SDK setup).

Note: JSON remains the final build artifact, but sample content is generated at prebuild time so it stays DRY and synchronized.

### Phase 3.5: Shared Types Visibility

1. Ensure API OpenAPI generation includes complete `components.schemas` for shared models/enums.
2. Ensure Fumadocs reference navigation exposes the schema/types area (do not hide it).

### Phase 4: Searchability and discoverability

1. Ensure operation pages are discoverable by SDK names:
   - include stable `operationId` aligned to SDK function names where appropriate.

2. Verify docs search index includes operation-page code tab content or equivalent indexed metadata.
3. Add acceptance test: searching `createAttestation` returns the corresponding reference operation page.

### Phase 5: Quality gates

1. Add link checks and “no duplicate page” checks in CI.
2. Validate docs build from repo root without manual services running.
3. Add redirects for removed SDK pages to avoid broken external links.
4. Add editorial checks for common violations (at minimum: `fide:` predicate usage, `did:fide:` formatting, RFC 3339 timestamps).

## Acceptance Criteria

- A new developer can find “how to call create attestation” in one obvious place.
- No duplicated reference content across `reference/*` and `sdks/*`.
- OpenAPI remains single source of truth for operation contract.
- Build is stable from root (`pnpm build`) without requiring local API process.
- Total authored long-form docs reduced to protocol + small guide set.
- Searching SDK function names (e.g. `createAttestation`) returns the relevant reference page.
- Shared types/schemas are discoverable in Reference without endpoint-specific navigation.
- SDK functions/types are discoverable without hand-written per-language trees.
- Protocol/Guides/Reference pages comply with the guardrails (no `fide:` predicates; `did:fide:` formatting in prose; required cross-links; valid placeholders; UTC timestamps).
