**Purpose**: User-created visual workflows that serve as reusable automation tools agents can discover and execute.

**Type**: **User Workflows** - On-demand, user-created workflows built via visual drag-and-drop builder

**Related**: See `TASK-DISPATCH-STRATEGY.md` for system dispatch (event-driven, automatic agent wake-up when tasks are assigned)

## References
- Uses Vercel Workflow DevKit / Workflow SDK patterns.
- Internal docs: `docs/PACKAGES/ai-sdk-docs`
- Reference repo docs: `docs/SUB-PROJECTS/_reference_only/workflow-orchestration/vercel-workflow-development-kit/docs`

## Current state
- Workflows are primarily treated as callable tools via the local-tool adapters (list/run/create/update).
- Storage is Supabase-backed (nodes/edges persisted) with RLS enforcement on access.
- No automatic task/project/milestone triggers are wired today; invocation is manual/explicit.
- Inputs typically include `workflow_id` and `team_id`; codegen is available but optional.

## Current usage patterns
- Used when multi-step automation or traceable runs are desirable (triage, notifications, reports, QA).
- Direct tool calls are used for single-step, low-latency actions.
- Outputs include run status/logs and any artifacts returned to the caller; domain updates are applied by the workflow if authored that way.

## Security posture (as practiced)
- Calls are expected to use RLS-enforced Supabase clients; team scoping is required on reads/writes.
- Permission checks precede dispatch; workflows should not bypass RLS.
- Inputs to external APIs are validated; sensitive data is avoided in logs.

## Reliability posture (targets/plans)
- Aim for idempotent external calls, retries with backoff, and clear error states per node.
- Compensation steps are considered for destructive operations (not universal yet).

## Common runbooks (in use or intended)
- Bug triage: severity, reproduction notes, logs, owner suggestion/assignment, summary post.
- Spec draft: goals/requirements/acceptance criteria draft + notification.
- Release checklist: changelog, tests/verifications, announcement draft, status post.
- Status report: roll up tasks/milestones, risks, next steps; post to comms.
- On-call handoff: summarize incidents/open tasks; notify next on-call.

## Tool adapters (current)
- `list_workflows`: returns workflows visible to caller (team/role filtered).
- `run_workflow`: accepts `workflow_id` (or template), `team_id`, optional `task_id/project_id/milestone_id`, payload; returns run_id/status.
- `generate_workflow_code`: available for codegen.
- Contracts observed: include team context; distinguish permission errors vs runtime failures; basic auditing of caller/run.

## Domain playbooks (planned/optional wiring)
- Tasks: could trigger on create/assign/status change for triage/QA/notifications; would store last_run_id on task.
- Projects: could run kickoff, cadence reports, risk checks; would store status run refs and avoid duplicate channel posts.
- Milestones: could validate deps/dates, notify on start, roll up on completion; would keep updates idempotent.

## Directory Structure

User workflows are organized under `lib/workflows/`:

```
lib/workflows/
├── workflow-executor.workflow.ts    # Tool workflow executor (Vercel Workflow)
├── workflow-actions.ts              # Server actions for workflow execution
├── workflow-builder.tsx             # Visual builder UI
├── workflow-store.ts                # Tool workflow state management
├── workflow-api.ts                  # Tool workflow API adapter
├── workflow-codegen*.ts             # Code generation
├── workflow-logging.ts              # Tool workflow execution logging
├── workflow-logger.ts               # Workflow logger utility
├── execute-workflow.ts              # Workflow execution wrapper
├── credential-fetcher.ts            # Tool workflow credential fetching
├── integration-adapter.ts           # Tool workflow integration adapter
├── integration-store.ts            # Tool workflow integration store
├── steps/                           # Tool workflow step handlers
│   ├── step-handler.ts
│   ├── http-request-step.ts
│   ├── database-query-step.ts
│   ├── condition-step.ts
│   └── fide-twin-step.ts
├── registry/                        # Tool workflow registries
│   ├── codegen-registry.ts
│   ├── step-registry.ts
│   └── output-display-configs.ts
└── utils/                           # Tool workflow utilities
    ├── redact.ts
    └── time.ts
```

**Key Files**:
- `fide/lib/workflows/workflow-executor.workflow.ts` - Executes user-created workflows (Vercel Workflow)
- `fide/lib/workflows/workflow-actions.ts` - Server actions for executing workflows from UI
- `fide/lib/workflows/workflow-builder.tsx` - Visual drag-and-drop builder UI
- `fide/lib/workflows/execute-workflow.ts` - Workflow execution wrapper (used by tools)
- `fide/lib/local-tools/workflows/runWorkflow.ts` - Agent tool for running workflows
- `fide/components/workflow/` - Workflow builder UI components

**Note**: User workflows are separate from system dispatch (`services/dispatch/`). System dispatch is event-driven and automatic (agent wake-up), while user workflows are on-demand tools that agents can discover and execute.

## Competitive Landscape

How Fide's user workflows compare to other workflow builders (see `_strategies/competitive-analysis/` for full details):

| Framework | Workflow Approach | Advancement | Action | Notes |
|-----------|------------------|-------------|--------|-------|
| **LangGraph** | StateGraph + Functional API, checkpointing, subgraphs | ⬆️ More Advanced | 🎯 Adopt | See `_strategies/competitive-analysis/langchain/ARCHITECTURE.md` |
| **LlamaIndex Workflows** | Event-driven async, typed events, self-correction | ⬆️ More Advanced | 👀 Monitor | See `_strategies/competitive-analysis/llamaindex/ARCHITECTURE.md` |
| **Mastra** | `.then()/.branch()/.parallel()` syntax, suspend/resume | ➡️ Comparable | 👀 Monitor | See `_strategies/competitive-analysis/mastra/ARCHITECTURE.md` |
| **CrewAI Flows** | Sequential/hierarchical processes | ➡️ Comparable | ⏭️ Skip | See `_strategies/competitive-analysis/crewai-inc/ARCHITECTURE.md` |
| **Atlassian Rovo Studio** | No-code agent builder with 100+ Skills | ⬆️ More Advanced | 👀 Monitor | See `_strategies/competitive-analysis/atlassian/ARCHITECTURE.md` |

**Fide's Current Position**:
- ✅ Visual drag-and-drop builder
- ✅ Supabase-backed storage with RLS
- ✅ Agent-callable as tools
- ✅ Step types: HTTP, database, conditions, Fide Twin
- ❌ No event-driven triggers (planned)
- ❌ No checkpointing/resume mid-workflow
- ❌ No workflow templates library (planned)

**Key Patterns to Consider**:
1. **Resumable Streams** (LangGraph): Pause for input, resume exact state
2. **Human-in-the-Loop** (Mastra): Suspend workflows for approval, resume from stored state
3. **No-Code Builder** (Rovo Studio): Pre-built skills for common patterns
4. **Typed Events** (LlamaIndex): Pydantic models for step communication
