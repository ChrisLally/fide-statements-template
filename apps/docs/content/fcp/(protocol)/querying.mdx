**Querying** is the Read Path of the Fide Context Protocol.
This page covers how applications retrieve, resolve, and score graph data.

Because FCP is distributed, querying is not just `SELECT *`. It follows a three-step flow:

1.  **Retrieve & Resolve**: Fetching statements and merging multiple IDs into one entity.
2.  **Consensus**: Aggregating trust signals to decide what is "true".
3.  **Traverse**: Following relationships to discover connected context.

---

## 1. Retrieve & Resolve [#resolve]

In FCP, an entity resolves to one [**Primary Fide ID**](/docs/identifiers#primary) under a chosen trust policy, while multiple external identifiers are linked as aliases through [Alias Clustering (`owl:sameAs`)](/docs/schema/relationships#sameas).

### Primary + Aliases Pattern [#primary-aliases]

To make sense of the chaos, FCP relies on two concepts:
1.  **[Primary Fide ID](/docs/identifiers#primary)**: A stable, protocol-native identifier (e.g. `did:fide:0x10...`) derived from a Genesis Statement.
2.  **[Aliases](/docs/schema/relationships#sameas)**: Other IDs (LinkedIn, X) linked via `owl:sameAs`.

<Callout type="info" title="Where to Learn the Link Pattern">
For the statement pattern that creates alias links, see [Relationships: Alias Clustering (`owl:sameAs`)](/docs/schema/relationships#sameas).
</Callout>

<Accordions>
<Accordion title="Deep Dive: How Resolution Works">

**The Problem**: Alice has 3 IDs (LinkedIn, X, Internal CRM).

**The Solution**:
An indexer maintains a mapping of **Alias → Primary** based on trust.

1.  **Ingest**: Collect statements linking Aliases to Primaries.
2.  **Resolve**: Decide which links to trust (filtering out spam/fake links).
3.  **Map**: Store a resolved index where `Alice_LinkedIn` points to `Alice_Primary`.

**Query Logic:**
To build a profile for "Alice", an indexer typically:
1.  Chooses a resolution policy (what `owl:sameAs` evidence it accepts).
2.  Materializes the resolved mapping in `fcp_statements_identifiers_resolved`.
3.  Queries the materialized view by canonical subject fingerprint.

```sql
-- The materialized view pre-resolves identifiers; query by canonical subject
SELECT *
FROM fcp_statements_identifiers_resolved
WHERE subject_fingerprint = '<alice_canonical_fingerprint>';
```

This makes querying simple: Ask for **one canonical fingerprint**, get **all context** (from LinkedIn, X, CRM) resolved into it.

## 2. Consensus (Scoring Trust) [#consensus]

Once you have the data, how do you know what is true?

FCP uses a **Subjective Trust Model**. The protocol provides the votes (Evaluations); **YOU** decide the count.

<Callout type="info" title="Consensus Depends on Evaluation Statements">
Consensus requires querying [evaluation statements](/docs/evaluating#issuing), not just base attributes/relationships.

- Evaluation statement shape: [Statement schema](/docs/schema#statement)
- Evaluation method semantics: [EvaluationMethod](/docs/entities#evaluation-method)
- Signal issuance flow: [Evaluating](/docs/evaluating#issuing)
</Callout>

### The Ground Truth Formula [#formula]

```
Ground Truth = Σ (Evaluator Verdict × Evaluator Weight)
```

| Value | Status | Meaning |
|-------|--------|---------|
| `> 0` | **Verified** | Positive agreement from trusted peers |
| `< 0` | **Refuted** | Negative findings (fraud, falsehood) |
| `= 0` | **Contested** | Conflicting opinions or no data |

### Weighting Strategies [#weighting]

Evaluator weights are **application-side logic**. You choose who to trust:

<Tabs items={['Reputation-Based', 'Transparency-Based']}>
<Tab value="Reputation-Based">
Trust is earned by history.

*   High Accuracy Peer = **1.0x**
*   New/Unknown Peer = **0.1x**
*   Known Spammer = **-1.0x**
</Tab>
<Tab value="Transparency-Based">
Trust is earned by openness.

*   [White-box Method](/docs/evaluating#methods) (Open Code) = **1.2x Boost**
*   Black-box Method (Hidden) = **0.8x Penalty**
</Tab>
</Tabs>

<Accordions>
<Accordion title="Code: Calculating Trust Scores">

This calculation happens at **Query Time**, enabling different apps to have different risk tolerances.

```typescript
async function calculateTrustScore(entityID) {
  const evaluations = await getEvaluations(entityID);
  
  let score = 0;
  for (const eval of evaluations) {
    // Recursive: How much do I trust the Evaluator?
    const evaluatorWeight = await getReputation(eval.signerID);
    
    // Add weighted verdict
    score += (eval.verdict * evaluatorWeight);
  }
  
  return score; // Normalized 0-100 or Verdict
}
```

## 3. API Patterns [#api]

FCP supports standard patterns for accessing the resolved graph.

### JSON-LD Response [#json-ld]
All responses use **JSON-LD** with standard vocabulary prefixes (`schema:`, `prov:`, `owl:`, `sec:`) to ensure interoperability.

```json
{
  "@id": "did:fide:0x10...",
  "schema:name": "Alice Johnson"
}
```

<Callout type="info" title="Computed Fields Are Indexer-Specific">
Consensus outputs (like a "trust score") are derived at query time from [Evaluations](/docs/evaluating), not standardized as a single core predicate in the protocol.
</Callout>
