FCP uses a **Standardized Graph Structure** — we break all data down into simple sentences called **Statements**. Universally, this follows the pattern: **Subject → Predicate (Relationship) → Object**.

<Callout type="info" title="Statement as an Entity">
A [Statement](/docs/entities#statement) is a protocol [Entity](/docs/entities#protocol).

When a Statement Fide ID (`did:fide:0x00...`) is calculated, its `rawIdentifier` is derived from the Statement's contents (canonical JSON), then hashed by [`calculateFideId`](/docs/identifiers#calculate).
</Callout>

<Callout type="info" title="SDK Implementation">
The SDK provides `createStatement()` and `buildStatementBatch()` helpers. See [Statement](/docs/sdks/js/statement) and [Schema](/docs/sdks/js/schema) in the SDK documentation for implementation examples.
</Callout>

This simple structure, combined with strictly typed entities (see [**Entities**](/docs/entities)), powers key patterns:

1.  **[Attributes](/docs/schema/attributes)**: "Alice (Subject) → has name (Relationship) → Alice Smith (Value)"
2.  **[Relationships](/docs/schema/relationships)**: "Alice → works for → Acme Corp"
3.  **[Actions](/docs/schema/actions)**: "Agent → performed → Action"

For specialized patterns involving protocol entities:
- **[Evaluations](/docs/evaluating)**: "Entity → EvaluationMethod → Score"
- **[Signed Statements](/docs/attesting)**: "Statement → prov:wasGeneratedBy → Attestation"

## The Statement Schema [#statement]

Every piece of data in FCP starts as an **unsigned assertion** (a Statement). When signed, it becomes a **Signed Statement** (linked to an Attestation).

**The Structure:**
*   **Subject**: Who or What is being described (e.g., "Alice").
*   **Predicate**: The relationship (e.g., "works for").
*   **Object**: The value or target (e.g., "Acme Corp").

### Technical Schema: FideStatement [#technical-schema]

The **FideStatement** is the base unit. Each Fide ID is calculated using [`calculateFideId`](/docs/identifiers#calculate).

Statement Fide IDs (`did:fide:0x00...`) are derived from the Statement's canonical JSON form — see [Statement](/docs/entities#statement).

## Anatomy of a Statement [#anatomy]

### Subject [#subject]
The **Subject** is always a valid [**Fide Entity**](/docs/entities). It can be:
*   [**Active Entities**](/docs/entities#active) ([Person](/docs/entities#person), [Organization](/docs/entities#organization))
*   [**Inactive Entities**](/docs/entities#inactive) ([Place](/docs/entities#place), [Product](/docs/entities#product), [CreativeWork](/docs/entities#creative-work))
*   [**Protocol Entities**](/docs/entities#protocol) ([Statement](/docs/entities#statement), [Attestation](/docs/entities#attestation))

This recursiveness allows FCP to be powerful—you can make statements about other statements.

### Predicate [#predicate]
The **Predicate** defines the relationship type. Must use **0x65** (CreativeWork+Product) or **0xe5** (EvaluationMethod+Product). FCP uses standard vocabularies (primarily **Schema.org**).

<Callout type="info" title="Shorthand in docs">
Docs use shorthand (e.g. `owl:sameAs`, `schema:name`) for readability. The SDK expands these to full URLs before hashing via `expandPredicateIdentifier`. Stored statements use canonical full URLs.
</Callout>

| Pattern | Concept | Example Predicate |
|----------|---------|-------------------|
| [**Attributes**](/docs/schema/attributes) | Name | `schema:name` |
| [**Actions**](/docs/schema/actions) | Status | `schema:actionStatus` |
| [**Attributes**](/docs/schema/attributes) | Type | `schema:type` |

<Accordions>
<Accordion title="Developer Note: Casing Conventions">

*   **Predicates** are `lowerCamelCase` (e.g., `schema:worksFor`).
*   **Entity Types** are `PascalCase` (e.g., `schema:Person`).
*   **Standard Predicates** use established namespaces: `schema:`, `prov:`, `owl:`, `sec:`.

**Note on Types**: The [Fide ID](/docs/identifiers#type) encodes the fundamental entity type (e.g. [Person](/docs/entities#person)). The `schema:type` predicate is used for explicit RDF typing and fine-grained sub-typing (e.g. `schema:Person`, `schema:SoftwareEngineer`).

</Accordion>
</Accordions>

## Supported Vocabularies [#vocabularies]

While **Schema.org** is the default vocabulary for FCP, sophisticated World Models often require higher precision for identity, lineage, and concept mapping. FCP explicitly supports and encourages the following standard namespaces:

### 1. OWL (Web Ontology Language) [#owl]

<Accordions>
<Accordion title="Strict Identity Logic">

**Use for:** Strict logical definitions and identity management.

* **`owl:sameAs`**: Unlike `schema:sameAs` (which can be loose), use this when two entities are mathematically or strictly identical. Indexers may choose to merge these nodes automatically depending on their trust and resolution policy.
* **`owl:differentFrom`**: Explicitly state that two entities are **not** the same (useful for resolving confusion between similar names).

</Accordion>
</Accordions>

### 2. SKOS (Simple Knowledge Organization System) [#skos]

<Accordions>
<Accordion title="Fuzzy Concepts & Hierarchies">

**Use for:** Defining concepts, categories, and "fuzzy" relationships where strict identity doesn't apply.

* **`skos:broader` / `skos:narrower`**: Use this for hierarchies.
  * _Example:_ `"Machine Learning" (Concept) → skos:broader → "Computer Science" (Concept)`

* **`skos:related`**: Use this for associative links.
  * _Example:_ `"Tennis" (Concept) → skos:related → "Racket" (Product)`

</Accordion>
</Accordions>

### 3. PROV-O (Provenance Ontology) [#prov-o]

<Accordions>
<Accordion title="Lineage & History">

**Use for:** Detailed lineage of **CreativeWorks** or **Immutable States**.

* **`prov:wasDerivedFrom`**: Indicates that a new version was built from an old one.
  * _Example:_ `DraftV2 → prov:wasDerivedFrom → DraftV1`

* **`prov:wasGeneratedBy`**: Links a static asset to the **Action** that created it.
  * _Example:_ `Report.pdf → prov:wasGeneratedBy → Action:GenerateReport`

* **`prov:wasRevisionOf`**: Indicates that a new entity replaces an old one (Corrections).
  * _Example:_ `StatementV2 → prov:wasRevisionOf → StatementV1`

* **`prov:wasInformedBy`**: Indicates that one activity led to another (Workflow DAGs).
  * _Example:_ `Action:ShipGoods → prov:wasInformedBy → Action:ReceivePayment`

</Accordion>
</Accordions>

### 4. W3C Security Vocabulary [#sec]

<Accordions>
<Accordion title="Security & ControlKeys">

**Use for:** Linking keys, agents, and controllers.

* **`sec:controller`**: The entity that holds the private keys for an account/agent.
  * _Example:_ `Wallet (did:fide:0x88...) → sec:controller → Alice (did:fide:0x15...)`
  * _Note:_ This aligns with the W3C DID specification for control relationships.

</Accordion>
</Accordions>

### Object [#object]
The **Object** completes the sentence and is always an **Entity** (a Fide ID).

In practice:
*   For **Relationships**, the object is often a non-`CreativeWork` entity (e.g. a [Person](/docs/entities#person)/[Organization](/docs/entities#organization)/[Product](/docs/entities#product)).
*   For **Attributes**, the object is typically a `CreativeWork` entity whose raw identifier is the value itself (e.g. `"Alice Smith"`).

See [Attributes](/docs/schema/attributes) for the recommended convention (predicate term + value are both `CreativeWork` entities).

<Callout title="Where is the Context?" type="info">
We often need to say *when* something happened or *how long* it is true. In FCP, context is simply an **Attribute** of the statement itself.

👉 **[Read the full Context & Timestamps guide](/docs/schema/attributes#context)**
</Callout>

<Callout title="Mistakes Happen: The Correction Pattern" type="info">
To correct a statement, we don't delete it. Instead, we **replace** it using the `prov:wasRevisionOf` relationship.

👉 **[Read the full Correction Pattern guide](/docs/schema/relationships#correction)**
</Callout>
