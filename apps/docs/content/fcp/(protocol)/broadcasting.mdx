**Broadcasting** is the Transport Path of the Fide Context Protocol.
This page covers how signed [**Attestations**](/docs/attesting) are published to distributed registries for discovery by indexers.

<Callout type="info">
**Broadcasting requires signed attestations** — see [**Attesting**](/docs/attesting) to understand the signature process before attestations can be broadcast.
</Callout>

## Broadcasting vs Direct Submission [#modes]

Unlike traditional databases where data is "submitted" to a central system, FCP uses **broadcasting**:

- **Broadcasting**: Publishing attestations to publicly discoverable, distributed locations
- **Indexing**: Independent services that discover, pull, and ingest these broadcasts
- **Sovereignty**: Users own their data repositories; the protocol just aggregates them

---

## Sovereign Broadcasting: Fide Attestation Registries [#registries]

The Fide Context Protocol currently defines Git-based Sovereign Registries as the first officially protocol-compliant broadcasting option. Additional compliant broadcasting options may be added in future versions.

<Callout type="info" title="SDK Implementation">
The SDK provides `formatAttestationForJSONL()`, `generateRegistryPath()`, and `generateJSONLFilename()` helpers. See [Broadcasting Guide](/docs/guides/broadcasting) in the SDK documentation for implementation examples.
</Callout>

| Mode | Description | Use Case |
|------|-------------|----------|
| **Sovereign Registry** | Organizations create their own public repositories | Maximum control, high-volume agents |
| **Shared Registry** | Community-operated repos accepting attestations via PR | Users who don't want to manage infrastructure |

## Discovery via Topics [#discovery]

To ensure Indexers can find your Sovereign Registry, you should signal its existence.

<Callout type="info">
**Topic Recommendation**: Add the topic `fide-attestation-registry` to your GitHub repository for automatic discovery by public indexers. This is **recommended but not required** — indexers can also be manually configured to sync from any Git repository.
</Callout>

**How Discovery Works:**
1. **Scan**: Public indexers scan GitHub by default for repos tagged with `fide-attestation-registry`
2. **Validate**: Check if the repo follows the valid directory structure (or has a `.fcp-registry.json` config)
3. **Sync**: Add to git fetch loop, automatically ingesting new attestations

Only GitHub repositories are officially supported for automatic discovery. GitLab, Bitbucket, and self-hosted Git are future considerations.

**Manual Configuration:**
Indexers can be manually configured to sync from any Git repository URL, regardless of topics or GitHub hosting. The topic is purely for convenience in automatic discovery.

## Private Registries [#private]

**Git permissions = access control.** FCP doesn't define protocol-level access control because Git already solves this.

### How Private Registries Work [#how-private-registries-work]

1. **Create a private Git repository** (GitHub, GitLab, Bitbucket, self-hosted)
2. **Grant access** to your trusted indexer via Git mechanisms:
   - SSH keys
   - Deploy keys (read-only recommended)
   - Personal access tokens
   - OAuth apps
3. **Configure your indexer** to sync from the private repo
4. **Attestations remain private** — only the indexer with Git access can read them

### No Protocol Changes Needed [#no-protocol-changes-needed]

The protocol is identical for public and private registries:
- Same attestation schemas
- Same JSONL line format
- Same directory structure
- Same [verifiable signatures](/docs/attesting)

**The only difference is who has read access to the Git repository.**

### Edge Cases (Application-Layer) [#edge-cases-application-layer]

Some privacy patterns are **not** protocol concerns:

| Pattern | Status | Approach |
|---------|--------|----------|
| **Field-level encryption** | Application-layer | Encrypt `attributeValue` before signing; indexer stores encrypted data |
| **Selective disclosure** | Application-layer | Use multiple registries with different access controls |
| **Zero-knowledge proofs** | Future consideration | Not in v1 scope |

<Callout type="info">
**GDPR / Right to be Forgotten**: Users control their Git repos. To delete attestations, rewrite Git history and force-push. Indexers should respect this (though cached data may persist temporarily).
</Callout>

## Timestamp Clarity [#timestamps]

Broadcasting is about **publication for discovery**. Any "first-seen" and ordering metadata is handled by indexers — see [Indexing](/docs/indexing).

<Callout type="warning" title="Trust Warning">
**Never rely on filename timestamps or Git commit times for validity.** Malicious actors can backdate files and repositories can rewrite history.
</Callout>

<Callout type="info" title="For Humans">
Developers can manually search GitHub for `topic:fide-attestation-registry` to discover public data sources.
</Callout>

This structure ensures reliable discovery while keeping ordering/first-seen semantics indexer-defined.

## Broadcasting Process [#process]

<Steps>
<Step>
### Setup [#setup]
Create a public repository on GitHub (or any Git host). 

**Optional but recommended:** Add the topic `fide-attestation-registry` for automatic discovery by public indexers.

**Optional:** Create a `.fcp-registry.json` file if you want to use a custom attestations path (defaults to `attestations/`).
</Step>

<Step>
### Preparation [#preparation]
Format signed attestations into the required JSONL/Directory structure locally. The SDK provides helpers for this - see [Broadcasting Guide](/docs/guides/broadcasting) for implementation examples.
</Step>

<Step>
### Publication [#publication]
Commit and push to your repository:

```bash
git add attestations/2024/01/15/
git commit -m "Broadcast daily attestations"
git push origin main
```
</Step>

<Step>
### Verification [#verification]
Your attestations are now live. Indexers scanning the `fide-attestation-registry` topic will discover your repo and begin ingestion. See [**Indexing**](/docs/indexing) for what happens next.
</Step>
</Steps>

<Accordion title="Fide Attestation Registry Structure">

For a Git repository to be recognized as a valid Fide attestation registry, it must follow this structure:

### Directory Hierarchy [#directory-hierarchy]

Attestations are organized chronologically. This `YYYY/MM/DD` structure allows indexers to efficiently sync only new days without re-scanning the entire repo:

**Default Structure:**
```
my-sovereign-registry/
├── attestations/          # Default location
│   ├── 2024/
│   │   ├── 12/
│   │   │   ├── 31/
│   │   │   │   ├── 2024-12-31-1400-1.jsonl
│   │   │   │   └── 2024-12-31-1400-2.jsonl
```

**Custom Location (via Config):**
You can specify a custom path using a `.fcp-registry.json` file at the repository root:

```json
{
  "attestationsPath": "data/fcp-attestations",
  "metadata": {
    "name": "My Registry",
    "description": "Custom attestations location"
  }
}
```

**Default Paths Checked (if no config):**
Indexers will check these locations in order:
1. `attestations/` (default)
2. `fcp-attestations/` (more explicit)
3. `.fcp/attestations/` (hidden, organized)

**Why Custom Paths?**
- Better developer experience: Keep attestations in a subdirectory if your repo contains other content
- Monorepo support: Organize attestations alongside other project files
- Flexibility: Choose a location that fits your repository structure

### Filename Convention [#filename-convention]

```
YYYY-MM-DD-{HHmm}-{sequence}.jsonl
```

Where:
- `YYYY-MM-DD`: Date of the broadcast window
- `{HHmm}`: Start time of the window (24-hour format, e.g., `1400` for 2:00 PM)
- `{sequence}`: Incrementing counter (1, 2, 3...) for rollover within the same time window
- `.jsonl`: JSON Lines format (each line is a complete [**Attestation**](/docs/attesting))

**Example:** `2024-01-19-1400-1.jsonl`

### Rollover Rules [#rollover]

To maintain performance and compatibility with Git hosts (e.g., GitHub's 100MB file limit):

1. **Size Limit**: Files **SHOULD** be rotated when they reach **50MB** in size
2. **Sequence Increment**: When rotating due to size within the same time window, increment the `{sequence}` number
3. **Append-Only**: Indexers expect files to be append-only. Never rewrite a published \`.jsonl\` file except for rare compliance/GDPR purges

**Example Rollover:**
- \`2024-01-19-1400-1.jsonl\` (reaches 50MB)
- \`2024-01-19-1400-2.jsonl\` (new file, same time window)
- \`2024-01-19-1400-3.jsonl\` (if needed)

### File Immutability [#file-immutability]

Once published, attestation files are **never modified**:
- **Corrections**: Made through new statements with [`prov:wasRevisionOf`](/docs/schema/relationships#correction)
- **Audit Trail**: Complete history preserved
- **Verification**: Anyone can replay the entire registry

<Accordion title="Trust & Safety">

### Distributed Model [#distributed-model]

**There is no central Fide Protocol Blacklist.** Each Indexer operator manages their own blocklists and trust policies.

| Layer | Responsibility |
|-------|----------------|
| **Git Host** | Platform moderation (CSAM/Terrorism/PII removal) |
| **Indexer** | Maintains blocklists of malicious repos |
| **User** | Responsible for content they host |

### Platform Compliance [#platform-compliance]

- Indexers respect platform "Takedowns" (repo deletion) as valid signals
- Git hosts may rewrite history via `git-filter-repo` for GDPR compliance
- Indexers should accept these "Compliance Rewrites" automatically
